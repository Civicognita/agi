# Impact Scoring Rules

**Document ID:** GOV-ISR-001
**Status:** THEORY |+.8|
**Version:** 0.1.0
**Effective:** 2026-02-20
**Authority:** #E0 (Wish Born)
**Review Cycle:** Every major version release or on QUANT value change request

---

## 1. Purpose

This document specifies the exact rules aionima uses to calculate, record, and query `$imp` (impactium) scores for every tracked interaction. It is the binding specification that governs:

- Which interactions are scored
- What numeric values are used
- How the formula is applied
- How historical alignment multiplies base scores
- Where scores are stored and how they are aggregated
- Who can change these rules and under what process

Implementors MUST treat this document as the source of truth. Where code diverges from this document, the document governs and the code is wrong.

---

## 2. Scope

These rules apply to all interactions processed by the aionima gateway and agent-bridge pipeline, specifically:

- All inbound channel messages from external entities (users, bots)
- All outbound responses generated by the agent
- System-level operations: task dispatch, verification requests, seal issuances, and skill invocations

Interactions that are NOT scored:

- Internal queue mechanics (enqueue, dequeue, retry)
- Health checks and gateway state transitions
- COA chain logging entries themselves (they are the audit record, not the scored event)
- Interactions from entities whose `verification_tier` is `'unverified'` and who have sent fewer than 3 prior interactions — these are queued but scoring is deferred until the third interaction, at which point all three are scored retroactively

## 3. The Formula

```
$imp = QUANT x VALUE[0BOOL] x (1 + 0BONUS)
```

This is the canonical 0SCALE formula from `core/0SCALE.md`. Each component is defined in the sections below. The computed `$imp` value is stored in `impact_interactions.imp_score`.

### 3.1 Component Summary

| Component | Type | Source | Description |
|-----------|------|--------|-------------|
| `QUANT` | `REAL` | Section 4 | Base measurement unit for the interaction type |
| `VALUE[0BOOL]` | `REAL` | Section 5 | Precision factor derived from the 7-point direction scale |
| `0BONUS` | `REAL` | Section 6 | Historical alignment multiplier, additive to 1.0 |
| `$imp` | `REAL` | Computed | Final score stored in `imp_score` column |

### 3.2 Signed Result

`$imp` is a signed real number. A positive score is a **0BOON** — the interaction contributed positively to the system. A negative score is a **0BURN** — the interaction was net harmful. A score of exactly 0 is neutral and adds nothing to any aggregate.

### 3.3 Implementation Reference

The formula is implemented in `packages/entity-model/src/impact.ts`, `ImpactRecorder.record()`:

```typescript
const value0bool = BOOL_VALUES[params.boolLabel];
const impScore = params.quant * value0bool * (1 + bonus);
```

The `BOOL_VALUES` constant maps each 0BOOL label to its `VALUE` factor. Changes to that constant are a governance event — see Section 7.

## 4. QUANT Values

QUANT is the base measurement unit assigned to each interaction type before any directional or alignment adjustment. It represents the raw weight of the interaction in the ledger.

### 4.1 Locked QUANT Table

These values are fixed. They do not change without a formal governance review (Section 7.2).

| Interaction Type | `work_type` Token | QUANT | Rationale |
|-----------------|-------------------|-------|-----------|
| Conversation — inbound message | `message_in` | 1 | Atomic unit: one message from an entity |
| Conversation — outbound response | `message_out` | 1 | Symmetric: one response costs equal weight |
| Task dispatch | `task_dispatch` | 3 | Multi-step coordination, higher cost/value |
| Verification request | `verification` | 5 | High-trust operation requiring identity confirmation |
| Seal issuance | `seal_issuance` | 10 | Permanent record event, highest base weight |
| Skill invocation | `tool_use` | 2 | Capability use; more than message, less than coordination |

### 4.2 Mapping to `work_type`

The `work_type` column in `impact_interactions` and `coa_chains` uses the tokens defined in the table above. These tokens are also declared in `packages/coa-chain/src/types.ts` as `COAWorkType`. When a new `COAWorkType` is added to the codebase, a corresponding QUANT entry MUST be added to this table before the type is deployed to production.

```typescript
// packages/coa-chain/src/types.ts — authoritative token list
export type COAWorkType =
  | "message_in"      // QUANT = 1
  | "message_out"     // QUANT = 1
  | "tool_use"        // QUANT = 2
  | "task_dispatch"   // QUANT = 3
  | "verification"    // QUANT = 5
  | "artifact"        // QUANT = 2  (treated as tool_use tier)
  | "commit"          // QUANT = 2  (treated as tool_use tier)
  | "action";         // QUANT = 2  (treated as tool_use tier)
```

Note: `artifact`, `commit`, and `action` were not initially enumerated in Section 4.1 because they share the `tool_use` QUANT tier. They are included here for completeness. Any divergence from the `tool_use` tier requires a governance vote.

### 4.3 Unknown Interaction Types

If an interaction arrives with a `work_type` that has no entry in this table, the gateway MUST:

1. Log a warning to the boot log with the unknown `work_type` value
2. Record the interaction with `quant = 1` (conservative default)
3. Set `value_0bool = 0` (neutral) regardless of classifier output
4. Flag the interaction with `channel = '__UNCLASSIFIED__'` so it can be audited

Unknown types are never silently scored at non-default values.

## 5. 0BOOL Scale — Direction Assessment

0BOOL determines whether an interaction was beneficial, neutral, or harmful. It maps to a VALUE precision factor that scales QUANT up or down — and determines the sign of `$imp`.

### 5.1 Scale Definition

The 7-point scale is canonical from `core/0SCALE.md` and `core/0TERMS.md`. The implementation lives in `BOOL_VALUES` in `packages/entity-model/src/impact.ts`.

| 0BOOL Label | Quant | VALUE Factor | State | aionima Meaning |
|-------------|-------|-------------|-------|-------------------|
| `0FALSE` | -3 | -1.0 | 0DOOM | Actively harmful: abuse, spam, coordinated manipulation, policy violation |
| `FALSE` | -2 | -0.5 | DOOM | Harmful: repeated boundary violations, bad-faith interactions, misinformation |
| `0-` | -1 | -0.25 | 0INDIFFERENT | Slightly negative: off-topic noise, ignored prompts, unproductive patterns |
| `NEUTRAL` | 0 | 0.0 | HARMONY | Default: no signal, new entity, unclassifiable, system-internal events |
| `0+` | +1 | 0.25 | FLOW | Slightly positive: engagement, genuine questions, cooperative behavior |
| `TRUE` | +2 | 0.5 | JOY | Positive: task completion, constructive feedback, skill use with good outcome |
| `0TRUE` | +3 | 1.0 | BLISS | Transformative: mission-critical coordination, seal events, major verified contributions |

Note: The aionima code uses `"0-"` where the core spec uses `-0`. These are the same value. The code spelling is authoritative for `BoolLabel` type matching.

### 5.2 Classification Method

Classification is a two-tier system:

**Tier 1 — Automated Rules (applied first)**

The gateway applies rule-based classification for interaction types where direction is deterministic. These rules require no LLM call.

| Condition | Assigned 0BOOL |
|-----------|----------------|
| `work_type = 'seal_issuance'` and entity `verification_tier = 'sealed'` | `0TRUE` |
| `work_type = 'verification'` with successful outcome | `TRUE` |
| `work_type = 'verification'` with failed outcome | `FALSE` |
| Entity on blocklist at time of interaction | `0FALSE` |
| `work_type = 'task_dispatch'` with confirmed completion | `TRUE` |
| `work_type = 'task_dispatch'` with failed or timed-out completion | `0-` |
| `work_type = 'message_out'` where entity did not respond within 24 hours | `0-` |
| Any interaction where message content matches abuse pattern rules | `0FALSE` |

Abuse pattern rules are maintained in `config/abuse-patterns.json`. Changes to that file are a governance event (Section 7.3).

**Tier 2 — LLM Assessment (applied when no Tier 1 rule matches)**

When no automated rule produces a classification, the agent bridge MAY request an LLM assessment. The LLM receives:

- The interaction type
- The message content (sanitized)
- The entity's 30-day interaction history summary (count by 0BOOL label only, not raw content)
- The scale definitions from this document (Section 5.1)

The LLM returns a `BoolLabel` and a confidence score (`|+value|`). If confidence is below `|+.6|` (below MAGIC threshold per `core/0TERMS.md`), the classification defaults to `NEUTRAL`. If confidence is `|+.6|` or above, the LLM classification is used.

The classification and confidence are stored in `impact_interactions` via the `work_type` field as a structured suffix (e.g., `message_in:llm:0+:0.8`) so audit queries can distinguish automated from LLM-assessed records.

**Tier 3 — Default**

If neither tier produces a result (agent bridge offline, LLM unavailable), all interactions default to `NEUTRAL`. This is fail-safe: a neutral score of 0 never inflates or deflates the ledger.

### 5.3 Override Mechanism

`#E0` may override any classification at any time by directly calling `ImpactRecorder.record()` with an explicit `boolLabel`. An override MUST:

1. Use a COA fingerprint with `#E0` in the entity segment (e.g., `$A0.#E0.@A0.C010`)
2. Include the original `impact_interactions.id` of the record being overridden in a corresponding `coa_chains` entry via the `ref` column
3. Result in a new `impact_interactions` row — the original row is never modified or deleted (ledger is forward-only)
4. The override row's `work_type` MUST be suffixed with `:override` (e.g., `message_in:override`)

Override rows are included in all aggregations. The net effect of an override is the sum of original score plus override score, which may cancel, amplify, or reverse the original.

## 6. 0BONUS Calculation

0BONUS is the historical alignment multiplier. It rewards entities whose past interactions have been consistently positive. It is additive to 1.0 in the formula — a 0BONUS of 0.0 means no multiplier effect; a 0BONUS of 0.5 means scores are 1.5x base.

### 6.1 Formula

```
0BONUS = SUM(positive_imp_scores) / BONUS_WINDOW_DIVISOR
```

Where:
- `positive_imp_scores` = all `imp_score` values > 0 from the entity's history within the lookback window
- `BONUS_WINDOW_DIVISOR` = 100 (fixed normalizer — see Section 6.3)
- Only positive scores contribute. Negative scores do not reduce 0BONUS directly; they simply do not add to it.

This matches the `core/0BONUS.md` definition: `0BONUS = SUM({<COI}) WHERE alignment > 0`.

### 6.2 Lookback Window

0BONUS is calculated from interactions within a **rolling 90-day window**. Interactions older than 90 days do not contribute to the current 0BONUS.

The `ImpactRecorder.getBalanceSince()` method provides the windowed sum. The caller computes:

```typescript
const since = new Date(Date.now() - 90 * 24 * 60 * 60 * 1000).toISOString();
const positiveSum = recorder.getPositiveBalanceSince(entityId, since);
const bonus = Math.min(positiveSum / BONUS_WINDOW_DIVISOR, BONUS_CAP);
```

Note: `getPositiveBalanceSince()` is a variant of `getBalanceSince()` that filters `imp_score > 0`. This method must be added to `ImpactRecorder` if not yet present.

### 6.3 Normalization Divisor

`BONUS_WINDOW_DIVISOR = 100`

An entity with 100 $imp of positive history in the window carries a 0BONUS of 1.0, meaning their current interactions score at 2.0x base. This divisor is a governance-locked value.

### 6.4 Cap

`BONUS_CAP = 2.0`

0BONUS is capped at 2.0. No entity may have their scores multiplied by more than 3.0x (1.0 base + 2.0 cap). This prevents runaway score inflation for long-established entities.

An entity with 200+ $imp of positive history in the window will have 0BONUS = 2.0, not higher.

### 6.5 New Entity Default

New entities with zero interaction history have `0BONUS = 0.0`. The multiplier term becomes `(1 + 0.0) = 1.0`, so the formula reduces to `$imp = QUANT x VALUE[0BOOL]`.

There is no provisional or goodwill bonus for new entities. Bonus must be earned through recorded history.

### 6.6 Decay

There is no separate decay function. The 90-day rolling window serves as natural decay: as old positive interactions age out of the window, 0BONUS decreases automatically. No decay coefficient is applied within the window.

## 7. $imp Ledger

### 7.1 Storage — `impact_interactions` Table

Every scored interaction produces exactly one row in `impact_interactions`. The table schema is defined in `packages/entity-model/src/schema.sql.ts`:

```sql
CREATE TABLE IF NOT EXISTS impact_interactions (
  id              TEXT NOT NULL PRIMARY KEY,          -- ULID
  entity_id       TEXT NOT NULL REFERENCES entities(id),
  coa_fingerprint TEXT NOT NULL REFERENCES coa_chains(fingerprint),
  channel         TEXT,                              -- e.g. 'telegram', 'discord'
  work_type       TEXT,                              -- COAWorkType token (+ optional suffix)
  quant           REAL NOT NULL,                     -- Section 4 QUANT value
  value_0bool     REAL NOT NULL,                     -- Section 5 VALUE factor
  bonus           REAL NOT NULL DEFAULT 0,           -- Section 6 0BONUS at time of mint
  imp_score       REAL NOT NULL,                     -- Computed: quant * value_0bool * (1 + bonus)
  created_at      TEXT NOT NULL                      -- ISO-8601
)
```

**Immutability:** Rows are never updated or deleted. The ledger is forward-only. Corrections are made by adding new rows (see Section 5.3, override mechanism).

**ID format:** Primary key is a ULID (26-character, sortable, monotonic). Use `ulid()` from the `ulid` package.

**COA linkage:** Every `impact_interactions` row MUST have a corresponding `coa_chains` row via `coa_fingerprint`. The COA record is the audit trail; the impact record is the score. Neither exists without the other.

### 7.2 Aggregation Queries

Standard aggregation patterns. All are expressed in terms of the `ImpactRecorder` API or raw SQL for dashboard consumers.

**Per-entity lifetime balance:**
```sql
SELECT COALESCE(SUM(imp_score), 0) AS balance
FROM impact_interactions
WHERE entity_id = ?
```
API: `ImpactRecorder.getBalance(entityId)`

**Per-entity rolling window balance (for 0BONUS calculation):**
```sql
SELECT COALESCE(SUM(imp_score), 0) AS balance
FROM impact_interactions
WHERE entity_id = ? AND created_at >= ?
```
API: `ImpactRecorder.getBalanceSince(entityId, since)`

**Per-channel aggregate (for channel health dashboard):**
```sql
SELECT channel,
       COALESCE(SUM(imp_score), 0)    AS total_imp,
       COUNT(*)                        AS interaction_count,
       COALESCE(AVG(imp_score), 0)    AS avg_imp
FROM impact_interactions
WHERE created_at >= ?
GROUP BY channel
ORDER BY total_imp DESC
```

**Per-domain aggregate (mapped through entity type):**
```sql
SELECT e.type AS entity_type,
       COALESCE(SUM(ii.imp_score), 0) AS total_imp,
       COUNT(*)                         AS interaction_count
FROM impact_interactions ii
JOIN entities e ON e.id = ii.entity_id
WHERE ii.created_at >= ?
GROUP BY e.type
ORDER BY total_imp DESC
```

**Interaction type breakdown for an entity:**
```sql
SELECT work_type,
       COUNT(*)                        AS count,
       COALESCE(SUM(imp_score), 0)    AS total_imp
FROM impact_interactions
WHERE entity_id = ?
GROUP BY work_type
ORDER BY total_imp DESC
```

### 7.3 Dashboard Display Requirements

Any UI surface displaying `$imp` scores MUST follow these conventions:

1. **Label:** Display as `$imp` — never as "points," "score," or "rating"
2. **Precision:** Display with 3 decimal places (`0.000`) — coarse interactions produce values like `0.250` or `-0.250`, not `0.25`
3. **Sign display:** Always show the sign explicitly: `+0.500`, `-1.000`, `0.000`
4. **Period:** Always state the aggregation period in the label (e.g., "90-day $imp balance" not just "$imp balance")
5. **0BOON / 0BURN split:** When showing lifetime totals, display both positive sum and negative sum separately in addition to net balance. Label positive sum as "0BOON" and negative sum as "0BURN"
6. **No rank normalization:** Never display a percentile or rank derived from $imp without first explicitly stating the comparison population and time period

## 8. Governance

### 8.1 Ownership

| Rule Area | Owner | Backup |
|-----------|-------|--------|
| QUANT values (Section 4) | #E0 | #O0 steering committee |
| VALUE/0BOOL mapping (Section 5.1) | #E0 | None — immutable without #E0 |
| Automated classification rules (Section 5.2, Tier 1) | #E0 or designated operator | Gateway maintainer |
| Abuse pattern rules (`config/abuse-patterns.json`) | Gateway maintainer | #E0 approval required |
| 0BONUS parameters: window, divisor, cap (Section 6) | #E0 | #O0 steering committee |
| Aggregation query definitions (Section 7.2) | Gateway maintainer | Any contributor via PR |
| Dashboard display conventions (Section 7.3) | UI maintainer | Gateway maintainer |

### 8.2 Change Process for QUANT Values

QUANT values are locked. The change process requires:

1. Open a proposal issue in the aionima repository tagged `governance:quant`
2. State the current value, proposed value, and quantitative rationale
3. The proposal must remain open for a minimum of 7 calendar days
4. Approval requires explicit written sign-off from #E0 in the issue thread
5. Implementation PR must reference the issue number and update both this document and `packages/coa-chain/src/types.ts` in the same commit
6. The commit COA fingerprint for the governance change must include `#E0` in the entity segment
7. The old value and new value are recorded in Section 9 (Revision History) of this document

No QUANT value change is valid without completing all 7 steps. A code change alone with no governance trail is invalid and must be reverted.

### 8.3 Change Process for Classification Rules

Automated classification rules (Tier 1, Section 5.2) and abuse patterns may be updated by the gateway maintainer without the 7-day window, subject to:

1. PR must be reviewed and approved by at least one other contributor
2. #E0 must be notified (via issue mention or direct message) within 24 hours of merge
3. #E0 retains the right to revert any classification rule change within 14 days without explanation
4. All changes are logged in Section 9 (Revision History)

LLM assessment prompts (Tier 2) follow the same process as classification rules.

### 8.4 Change Process for 0BONUS Parameters

Changes to `BONUS_WINDOW_DIVISOR`, `BONUS_CAP`, or the lookback window duration (Section 6) follow the same process as QUANT changes (Section 7.2) — 7-day window, #E0 approval, dual update of code and document.

### 8.5 Audit Trail Requirements

Every governance-relevant change MUST have a COA chain entry that satisfies:

- `resource_id`: The gateway or CI resource making the change (`$A0`, `$A1`, etc.)
- `entity_id` + `entity_alias`: The #E or #O entity responsible (not a worker or session alias)
- `work_type = 'commit'`
- `ref`: The git commit SHA of the change
- `payload_hash`: SHA-256 of the changed file content at the time of commit

The COA fingerprint for governance commits serves as the immutable audit record linking the human decision to the code change. Governance commits without a valid COA entry are flagged in the next monthly review.

### 8.6 Prohibited Changes

The following changes are NEVER permitted, regardless of who requests them:

- Deleting or modifying existing rows in `impact_interactions` (ledger is immutable)
- Setting `BONUS_CAP` above 4.0 (caps exist to prevent score manipulation)
- Setting `BONUS_WINDOW_DIVISOR` below 20 (prevents runaway inflation)
- Assigning `0TRUE` or `0FALSE` via automated Tier 1 rules to `message_in` or `message_out` interaction types without #E0 explicit approval — conversation-level extremes require human judgment
